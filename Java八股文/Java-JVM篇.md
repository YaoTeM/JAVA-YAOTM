# 一、运行时内存（结构 + 各区域作用）

## 1. 线程私有

| 区域                 | 作用                                               | 异常                                                         |
| -------------------- | -------------------------------------------------- | ------------------------------------------------------------ |
| **程序计数器（PC）** | 当前线程执行的字节码行号                           | 唯一不会 OOM                                                 |
| **虚拟机栈**         | 方法栈帧：局部变量表、操作数栈、动态链接、返回地址 | StackOverflowError（栈深度）、OutOfMemoryError（栈扩展失败） |
| **本地方法栈**       | Native 方法                                        | 同栈                                                         |

## 2. 线程共享

| 区域                                    | 作用                                   | 说明                                                         |
| --------------------------------------- | -------------------------------------- | ------------------------------------------------------------ |
| **堆（Heap）**                          | 对象实例、数组                         | 主 GC 区域，-Xms/-Xmx 等                                     |
| **方法区（JDK8 起为元空间 Metaspace）** | 类元信息、常量、静态变量、JIT 编译代码 | 不在堆里，-XX:MetaspaceSize 等，OOM 多为类太多或常量池/反射多 |

## 3. 直接内存（Direct Memory）

- NIO 使用的堆外内存，受 -XX:MaxDirectMemorySize 限制，Full GC 时也会回收（Cleaner），溢出时抛 OOM，堆 dump 可能看不出明显异常。

**和项目的联系**：RAG 请求会带大上下文、多 chunk、流式缓冲；线程池多、栈帧不深但线程数多会占栈；若用 NIO/Netty 等会占直接内存。调优时关注堆大小、线程栈、元空间和直接内存上限。

# 二、类加载与双亲委派

## 1. 加载过程（三阶段）

- **加载**：读字节流、生成 Class 对象、可来自 class 文件、jar、网络等。
- **链接**：验证、准备（静态变量默认值）、解析（符号引用→直接引用）。
- **初始化**：执行 \<clinit\>，静态变量赋值、静态块。

## 2. 双亲委派

- **定义**：类加载请求先交给**父加载器**，父有就父加载，父没有子才尝试加载。
- **层次**：Bootstrap → Extension → Application（系统类路径），自定义类加载器父一般是 Application。
- **作用**：保证核心类（如 java.lang.*）只由 Bootstrap 加载，避免被应用类替换，保证安全与稳定。

## 3. 破坏双亲委派（了解即可）

- **SPI**：如 JDBC 的 Driver 由 Bootstrap 加载，实现类在应用 classpath，需要「线程上下文类加载器」由应用类加载器去加载实现类。
- **热部署/隔离**：OSGi、Tomcat 等用自定义类加载器做模块隔离、不同版本同名的类不冲突。

面试能说清：双亲委派是什么、为什么这样设计、SPI 为什么要用上下文类加载器即可。

# 三、常见 GC 算法与收集器（简要）

## 1. 算法

| 算法          | 思路                               | 缺点                   |
| ------------- | ---------------------------------- | ---------------------- |
| **标记-清除** | 标记存活，清除未标记               | 碎片化                 |
| **标记-整理** | 标记后把存活对象移到一端，清边界外 | 移动成本，停顿相对长   |
| **复制**      | 存活对象复制到另一块，整块回收原区 | 占一半空间，适合年轻代 |

## 2. 分代

- **年轻代**：Eden + 2 个 Survivor，多数对象朝生夕死，Minor GC 频繁、停顿短。
- **老年代**：存活久的对象，Major/Full GC 次数少但停顿长。

## 3. 常见收集器（会选型即可）

| 收集器                     | 特点                                        | 适用                                  |
| -------------------------- | ------------------------------------------- | ------------------------------------- |
| **Serial**                 | 单线程，STW                                 | 单核、小堆、客户端                    |
| **Parallel（吞吐量优先）** | 多线程，关注吞吐                            | 批处理、可接受较长 STW                |
| **CMS**                    | 并发标记清除，减少 STW，有碎片              | 老年代、低延迟，JDK9 后逐渐被 G1 替代 |
| **G1**                     | 分区、可设 MaxGCPauseMillis，兼顾吞吐和停顿 | 通用服务端，大堆、可控停顿            |
| **ZGC/Shenandoah**         | 超低延迟、大堆                              | 超大堆、对停顿极敏感                  |

**和项目**：RAG 服务对首包延迟、流式体验敏感，可选 G1 或 ZGC，设 -Xmx、-XX:MaxGCPauseMillis 等；若偏吞吐可考虑 Parallel。

# 四、JVM 排查与调优思路

## 1. 常用参数（口述级别）

- **堆**：-Xms/-Xmx 一致避免动态扩缩，-Xmn 或 -XX:NewRatio 年轻代大小。
- **栈**：-Xss 每线程栈大小，线程多时注意总栈占用。
- **元空间**：-XX:MetaspaceSize、-XX:MaxMetaspaceSize，避免类加载多导致 Metaspace OOM。
- **GC 日志**：-Xlog:gc* 或 -XX:+PrintGCDetails（视 JDK 版本），便于看停顿和原因。
- **OOM 时 dump**：-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=...，便于事后分析。

## 2. 排查思路

- **OOM**：看 dump，是堆还是 Metaspace；堆里看是哪种对象多（大数组、大集合、线程栈、本地缓存等）。结合业务：RAG 里可能是大上下文、缓存未限制、线程池队列堆积等。
- **CPU 高**：先看是 GC 线程还是业务线程；业务线程用 jstack 看栈，是否死循环、锁竞争、大量计算。
- **停顿/延迟**：看 GC 日志，是否 Full GC 或 long pause；再考虑调堆、换收集器、减大对象/缓存。

## 3. 和 Ragent 项目的结合点

- **多线程池**：线程多，-Xss 不宜过大，避免总栈占用过高。
- **大对象/大上下文**：检索结果、对话历史、流式缓冲可能造成临时大对象，关注年轻代和晋升，避免老年代暴涨触发 Full GC。
- **缓存**：若本地缓存（如意图树）过大，可限制大小或 TTL，减轻堆压力。
- **目标**：保证首包和流式延迟，可设 -XX:MaxGCPauseMillis（G1）或选用 ZGC，再配合堆大小和监控。

# 五、面试 Q&A

**Q1：JVM 运行时内存分为哪几块？堆和方法区（元空间）有什么区别？**  
答：线程私有有程序计数器、虚拟机栈、本地方法栈；线程共享有堆和方法区（JDK8 后方法区用元空间实现，在本地内存）。堆主要放对象实例和数组，是 GC 的主战场；方法区/元空间放类元信息、常量、静态变量等，不参与普通堆 GC，由 -XX:MetaspaceSize 等控制，类加载过多或反射多可能导致 Metaspace OOM。

---

**Q2：双亲委派是什么？为什么要这样设计？**  
答：双亲委派指类加载时先把请求交给父加载器，父能加载就父加载，父加载不了子加载器再尝试。这样能保证像 java.lang.* 这样的核心类只由 Bootstrap 加载，不会被应用里的同名类替换，保证安全与一致性。破坏双亲委派的典型是 SPI：核心 API 在 Bootstrap 里，实现类在应用 classpath，需要线程上下文类加载器让应用类加载器去加载实现类。

---

**Q3：常见的 GC 算法和收集器？你们服务更关注延迟还是吞吐？**  
答：算法有标记-清除、标记-整理、复制；年轻代多用复制，老年代多用标记-清除或标记-整理。收集器里 Serial 单线程；Parallel 吞吐优先；CMS 并发标记清除、减少停顿；G1 分区、可设目标停顿时间；ZGC 超低延迟。我们 RAG 服务对首包和流式延迟敏感，更关注停顿，会优先考虑 G1 或 ZGC，并设合理的 -Xmx 和 MaxGCPauseMillis；若偏批处理或离线任务会更关注吞吐用 Parallel。

---

**Q4：发生 OOM 你会怎么排查？**  
答：先看是堆 OOM 还是 Metaspace OOM。堆 OOM 时用 -XX:+HeapDumpOnOutOfMemoryError 打出 dump，用 MAT 或 JProfiler 看哪些对象占得多、是否泄漏（如大 List、缓存未限制、线程池队列堆积）。结合业务：我们有多线程池、大上下文和检索结果，会重点看是否大对象过多、缓存无限增长或任务堆积。若是 Metaspace OOM，看是否类加载过多、反射或动态代理过多，必要时调大 MaxMetaspaceSize 或优化类加载。

---

**Q5：CPU 占用很高怎么排查？**  
答：先区分是 GC 线程还是业务线程。若是 GC 线程占比高，看 GC 日志是否频繁 Full GC 或长时间 GC，再考虑调堆、换收集器或减大对象。若是业务线程高，用 jstack 抓栈，看是否死循环、锁等待、大量计算或阻塞；我们项目里检索、模型调用、流式处理都可能耗 CPU，会结合 trace 和线程名定位到具体模块再优化。

---

**Q6：你们项目在 JVM 或调优上做过什么？**  
答：我们有多线程池和 TTL 透传，线程数较多，会注意 -Xss 不要设得过大，避免总栈占用过高。RAG 请求会带大上下文和检索结果，关注年轻代大小和晋升，避免老年代过快增长导致 Full GC。对延迟敏感，部署时会设 -Xms/-Xmx 一致、选 G1 或 ZGC，并开 GC 日志和 OOM 时 dump，便于线上排查。具体数值会按压测和监控再调。
