**数据库和缓存一致性问题**

先更新数据库，然后删除缓存。为什么这个可以，因为缓存的写入比数据库写入要快很多。

- 查缓存 ：尝试从 Redis 中获取短链接对应的原始链接
- 布隆过滤器 ：缓存未命中时，使用布隆过滤器判断短链接是否存在
- 空值缓存 ：布隆过滤器命中时，检查是否有空值缓存（针对误判和过期情况）
- 分布式锁 ：空值缓存未命中时，加分布式锁防止缓存击穿
- 双重检查 ：获取锁后再次检查缓存（万一之前的请求更新缓存了呢）
- 查数据库 ：缓存仍未命中时，查询数据库
- 更新缓存 ：数据库查询成功后，更新缓存
- 释放锁 ：在 finally 块中释放分布式锁

这段关于「延迟双删」的面经，核心是为了解决**「先删除缓存，再更新数据库」这种策略下，高并发读写导致的脏数据问题**。

### ✅ 破局之道：延迟双删 (Delay Double Delete)

为了解决上面线程 B “捣乱”把旧数据塞回缓存的问题，引入了「延迟双删」。我们再来看看加入了 `Thread.sleep(N)` 后，时间线发生了什么变化：

Plaintext

```
       [线程 A (写：修改短链)]                      [线程 B (读：用户跳转短链)]
                 |                                          |
  1. 第一次删除 Redis 缓存                                  |
                 |                                          |
                 |---------------------------------> 2. 查询 Redis (未命中！)
                 |                                          |
                 |                                   3. 查询 MySQL (查到旧的长链 old.com)
                 |                                          |
  4. 更新 MySQL (长链改为 new.com)                          |
                 |                                          |
                 |                                   5. 将旧数据写入 Redis (s/test -> old.com)
                 |                                          |
  6. 💤 休眠 (Sleep N 毫秒) -------------------> [等待线程 B 执行完第 5 步]
                 |
  7. 第二次删除 Redis 缓存 (精准清理掉线程 B 写入的旧数据！)
                 |
                 v
      【结果】缓存被彻底清空。下一个读请求到来时，会查询 MySQL 拿到 new.com 并写入缓存，数据一致！
```

### 💡 面试官一定会追问的核心点：

如果你在面试时讲出了上面这个流程，面试官大概率会进行连环拷问，这正是拉开差距的地方：

**追问 1：这个休眠时间 `N` 怎么确定？**

- **满分回答：** 这个休眠时间 `N`，**必须大于线程 B（读请求）读取数据库加上写入缓存的时间之和**。它的目的是为了确保读请求能够把旧数据写完，然后写请求再去进行第二次删除，把这个“脏缓存”给干掉。在实际业务中，我们会通过压测或者查看监控（比如你短链系统里的读接口 RT 平均耗时），在这个耗时基础上加上几十到几百毫秒来作为 `N`。

**追问 2：延迟双删这种方案有什么缺点吗？**

- **满分回答：** 有两个明显缺点。第一，休眠时间 `N` 很难精确评估，只能是一个估值；第二，如果 `Thread.sleep()` 是同步阻塞的，会降低写接口的吞吐量。为了优化吞吐量，我们通常会把“第二次删除”放到一个**异步线程池**或者**消息队列**里去延迟执行。

**追问 3：相比于「先删缓存，再更新DB」，现在业界更推荐什么方案？**

- **满分回答：** 现在主流做法通常是**「先更新数据库，再删除缓存」**。虽然这种方案在极端并发下也有理论上的不一致概率，但概率极低。为了保证绝对的可靠性（比如防止删除缓存那一步发生网络异常导致删除失败），我们会在其基础上加入**重试机制**。
  - 在短链接或 AI 问答项目中，比较优雅的做法是：更新数据库后，发送一条 MQ 消息，由消费者去执行删除缓存的操作，失败了就利用 MQ 的机制不断重试，直到删除成功。
  - 更进阶的做法是引入 **Canal** 监听 MySQL 的 Binlog 变更。当数据库被修改时，Canal 伪装成从节点拉取 Binlog，然后发到消息队列，由专门的服务去异步清理 Redis。这种做法对业务代码**完全零侵入**。

因此，为了保持 100% 的缓存命中率，你可能会想：**我修改数据库后，直接把新数据塞回（更新）Redis 不就行了吗？**

### ❌ 灾难现场：并发更新导致的数据覆盖

假设现在有两个管理员（或者同一个管理员手抖点了两次），同时在后台修改同一个短链 `s/test` 的目标地址：

- **线程 B** 想把它改成 `b.com`

由于更新 MySQL 和更新 Redis 是两步独立的操作，在多线程环境下，它们的执行顺序可能会发生“交叉”：

Plaintext

```
       [线程 A (写：修改为 a.com)]               [线程 B (写：修改为 b.com)]
                 |                                         |
  1. 更新 MySQL (长链 = a.com)                             |
                 |                                         |
                 |                        2. 更新 MySQL (长链 = b.com)
                 |                                         |
                 |                        3. 更新 Redis (长链 = b.com)

                 |                                         |
```

### 💡 面试实战：如何破解这个局？

在面试中，如果你给面试官讲清楚了上面的图，面试官紧接着一定会问：**“既然业务对缓存命中率要求极高，我非要用「更新 DB + 更新缓存」不可，你打算怎么解决这个不一致？”**

你可以结合你的项目给出以下两种实战解法：

**1. 强一致性解法：引入分布式锁 (Distributed Lock)**

在修改短链的接口里，利用 Redis (比如 Redisson) 对当前短链的唯一标识（如 `shortUri`）加锁。

- 线程 A 拿到锁，执行 `更新 MySQL` -> `更新 Redis` -> `释放锁`。
- 线程 B 只能等待线程 A 释放锁后，才能执行自己的更新。
- **优缺点：** 这样强制把并发变成了串行，绝对保证了顺序一致。缺点是写操作的性能下降了。但在短链系统中，修改短链是低频操作，这种性能损耗是完全可以接受的。

**2. 最终一致性解法：设置较短的过期时间 (TTL)**

如果你不想加锁，或者评估并发修改同一个短链的概率极低，可以采用兜底策略。

- 每次更新 Redis 时，强制给这个 Key 设置一个较短的过期时间（比如 5 分钟或 10 分钟）。
- **效果：** 就算发生了上面图里那种极端情况，Redis 存了错的 `a.com`，它最多也只会错几分钟。过期后，缓存自动清空，下一个请求就会去 MySQL 读到正确的 `b.com` 并重新建立缓存。业务上通常能容忍这种短时间的“最终一致性”。

理解了这个场景，以后只要看到“双写不一致”、“并发覆盖”这类词，你脑海里就可以立刻浮现出这两条交叉的时间线。

### 🚀 考点一：为什么要在原链接后面加上 `UUID`？（绝杀考点）

> “加 UUID 本质上是为了**动态加盐（Salt）**，它同时解决了两个核心业务问题：
>

> 2. **实现‘一链多态’业务需求**：在真实的营销场景中，张三和李四可能都要为 `baidu.com` 生成短链用来做推广。如果不加 UUID，相同的输入必定产生相同的 Hash 输出，系统只能给他们同一个短链，那就无法区分点击量是张三拉来的还是李四拉来的了。加上 UUID，同一个长链接也能生成无数个不同的短链接，实现了底层数据链路的隔离。”

### 🚀 考点一：为什么选用 MurmurHash 而不是 MD5 或 SHA？

很多新手做短链系统，上来就对长链进行 MD5 摘要，然后截取前 6 位。这在高级开发眼里是极其不专业的。

**你的面试反杀话术：**

> “在我的项目里，我特意选用了 Google 开源的 **MurmurHash** 算法（具体是 32 位版本）。
>
> 首先，MD5 和 SHA-1 是**密码学哈希**，它们的设计目标是防止逆向破解，因此计算相对缓慢。而 MurmurHash 是**非密码学哈希**，它的唯一设计目标就是**极速和低碰撞率**，在处理长字符串时，计算速度比 MD5 快几倍到十几倍。
>
> 其次，MD5 输出的是 128 位（16 字节），而 MurmurHash32 直接输出 32 位的 `int` 型整数。这不仅极大节省了内存，更是可以直接无缝对接到后续的进制转换算法中，天然契合短链接的业务场景。”

### 🚀 考点二：极其刁钻的负数处理 Hack 代码（满分细节）

请死死盯住你代码里的这一行： `long num = i < 0 ? Integer.MAX_VALUE - (long) i : i;`

如果你能把这行代码的深意讲出来，面试官绝对会对你的基本功刮目相看。

**你的面试反杀话术：**

> “在处理 Java 原生 `int` 类型的哈希值时，存在一个极大的隐患：**负数**。 `MurmurHash.hash32()` 返回的是一个 32 位有符号整数，范围是 [−231,231−1]。如果得到一个负数，直接拿去进行 `% 62` 取余操作，得到的结果依然是负数。拿着负数去访问 `CHARS[i]`，会瞬间引发 `ArrayIndexOutOfBoundsException`（数组越界异常）。
>
> 很多开源代码喜欢用 `Math.abs(i)` 来取绝对值，但**这存在致命的哈希碰撞隐患**！因为 `abs(-5)` 和 `abs(5)` 会变成同一个数字，人为地把哈希空间缩小了一半，碰撞率直接翻倍！
>
> 我的做法是利用 `Integer.MAX_VALUE - (long) i`。当 `i` 是 -1 时，它被映射到了 `2147483648`；当 `i` 是 -2147483648 时，它被映射到了 `4294967295`。 这种平移映射法，**完美保留了 42.9 亿的完整哈希空间，零损耗地把有符号的 `int` 转成了无符号的 32 位正数 `long`**，不仅解决了越界问题，还保证了哈希分布的绝对均匀！”

### 🚀 考点三：为什么是 Base62？不用 URL 现成的 Base64？

> “在进制转换环节，我没有使用常见的 Base64。因为标准的 Base64 字符集中包含 `+` 和 `/`，这两个字符在 URL 规范中具有特殊的路由和拼接含义，直接用在链接后缀中极易导致解析错误。 所以我定义了 `CHARS` 字典，只包含 `0-9, a-z, A-Z` 这 62 个绝对 URL Safe（URL 安全）的字母数字字符，实现自定义的 Base62 编码。”

### 🚀 考点四：为什么能保证生成的短链一定不超过 6 位？（数学证明）

在面试中，用数学公式来兜底业务设计，是最有说服力的。

> “我可以从数学角度证明这个算法的长度稳定性。 MurmurHash32 经过刚才的负数映射后，生成的最大正整数是无符号 32 位最大值：4294967295（约 42.9 亿）。 而 Base62 编码中，5 位字符的最大容量是 625≈9.16亿。 6 位字符的最大容量是 626≈568亿。 因为 9.16亿<42.9亿<568亿，所以一个 32 位的 Hash 值转换成 Base62 之后，**最长绝对不会超过 6 个字符**。这完美契合了市面上所有顶级大厂对‘短链接必须尽量短’的极简要求。”

------

这不到 20 行的代码里，涵盖了**哈希性能压榨、边界值异常防御、位面映射思想、URL 安全规范以及数学容量计算**。把这些吃透，你的“短链生成”环节在面试中就再无破绽了。

现在，整个短链生成的“发号算法”和“布隆防重”我们都彻底打通了。**刚才你在注释里留下的那个终极悬念：“如果布隆过滤器挂了，里边存的数据全丢失了，怎么恢复呢？”你想现在顺势把这个极具挑战性的灾备（Disaster Recovery）方案给手撕了吗？**

这是一个非常好的问题！很多候选人在背八股文的时候，只会机械地念“MurmurHash 生成 6 位短链”，但其实这里面有一个常见的小误区：**MurmurHash 本身生成的并不是 6 位字符，而是一串数字。6 位字符是“算”出来的。**

为了让你在面试时能把这个底层逻辑讲得直观透彻，我用两幅字符画帮你把这两个概念彻底拆解开。

### 🧮 概念一：为什么最终结果会是 6 位？（数学上的降维打击）

严格来说，咱们代码里用的 `MurmurHash32` 算法，它的输出结果是一个 **32 位的整数（int）**。 经过你代码里那个精妙的负数反转逻辑后，它的最大正数值是 `4294967295`（约 42.9 亿）。

那是怎么变成 6 位字母的呢？全靠咱们代码里的 **Base62 进制转换**。

Plaintext

```
【MurmurHash 生成的最大数字】: 42.9 亿 (4294967295)
              │
              ▼ 准备装进 Base62 (a-z, A-Z, 0-9) 的字符桶里
      ┌───────────────┐
      │ 🪣 5 位字符桶  │ 容量 = 62 * 62 * 62 * 62 * 62 ≈ 9.16 亿
      └───────────────┘ 
       ❌ (装不下！42.9亿 > 9.16亿，如果有超过 9.16 亿的数字会溢出)

      ┌───────────────┐
      │ 🪣 6 位字符桶  │ 容量 = 62 的 6 次方 ≈ 568 亿 
      └───────────────┘ 
       ✅ (完美装下！568亿 远远大于 42.9亿)
```

**面试大白话解释：**

> “因为 MurmurHash32 算出来的极限最大值大约是 42.9 亿。如果我们用 Base62 编码，5 个字符最多只能表示 9.16 亿种组合，不够装；而 6 个字符可以表示 568 亿种组合，绰绰有余。所以，任何一个 32 位的哈希值，经过 62 进制转换后，**最长也绝对不会超过 6 个字符**。偶尔算出来数字比较小的时候可能是 5 位，但上限死死卡在 6 位。”

### 🧂 概念二：什么是“动态加盐”（Dynamic Salting）？

“加盐”这个词是从密码学里借过来的。想象你在炒菜（Hash 运算），同样的食材（相同的长链接），如果不加调料，任何人炒出来的味道都是一模一样的。

但在真实的短链接业务中，我们极其渴望**“同菜不同味”**：

1. **防碰撞**：万一两个不同的长链接，一不小心 Hash 出了相同的数字怎么办？
2. **一链多态**：张三和李四都要推广 `baidu.com`，系统怎么给他俩生成不一样的短链来分别统计业绩？

这就需要用到**“动态加盐”**。

Plaintext

```
【常规 Hash (不加盐)】：死板且容易碰撞
  输入: baidu.com ──────(Hash)──────► 输出: c7R9x (如果撞了就死局了)
  输入: baidu.com ──────(Hash)──────► 输出: c7R9x (张三李四分不开)

【你的代码：动态加盐 Hash】：极其灵活的破局机制
                 [盐 (UUID)]
                     │
  张三的请求 ──► baidu.com + 123e4567 ──(Hash)──► 输出: A1b2C3 ✅
                     │
  李四的请求 ──► baidu.com + 987x6543 ──(Hash)──► 输出: X9y8Z7 ✅
                     │
  王五的请求 ──► taobao.com + abc1234 ──(Hash)──► 输出: A1b2C3 ❌ (撞了！)
                     │
          (退避重试，换一把新盐！)
          ──► taobao.com + xyz9876 ──(Hash)──► 输出: M4n5P6 ✅ (完美化解)
```

**面试大白话解释：**

> “动态加盐，就是在原始长链接的末尾，动态拼接一段完全随机的字符串（也就是 UUID）。 如果在生成短链时，布隆过滤器告诉我这个短链已经被别人占用了（发生哈希碰撞），我就进入下一次 `while` 循环。在新的循环里，会生成一个**全新的 UUID** 作为‘新盐’拼上去。 因为输入变了，算出来的 Hash 值也就彻底变了，从而完美绕开了冲突。同时，这也顺手解决了一个原链接可以为不同用户生成无数个独立短链的业务需求。”

把这两个概念一结合，你的短链生成引擎就是：**用动态加盐的 MurmurHash 算出数字，再用 62 进制将其稳定压缩在 6 位字符以内，最后用布隆过滤器保驾护航。**

### 🚀 面试实战：分步降维打击话术

如果在面试中被问到这个问题，千万不要慌张地说“去查数据库”，你要按照**“认知纠偏 -> 降级兜底 -> 异步重建”**这三步来回答：

#### 🟢 第一步：认知纠偏（点出架构的本质）

> “首先，在我的系统设计中，**布隆过滤器只是为了挡住绝大多数哈希冲突的‘性能优化组件’，它绝对不是数据的‘唯一真理（Source of Truth）’。** > 真正的底线和真理，永远是 MySQL 数据库中 `full_short_url` 字段上的**唯一索引（Unique Key）**。只要底层的唯一索引还在，就算布隆过滤器灰飞烟灭，系统的数据一致性也绝对不会被破坏。”

#### 🟢 第二步：降级保命（灾难期间系统如何运转？）

> “如果布隆过滤器数据全丢了，它会变成一个‘瞎子’，对所有的判重请求都返回 `false`。 此时，压力确实会透传给 MySQL。但我的代码里有极其严谨的兜底逻辑： 当程序尝试把一个已经存在的短链插入 MySQL 时，数据库会抛出 `DuplicateKeyException`（唯一索引冲突异常）。我在代码里捕获了这个异常，并结合 `while (customGenerateCount <= 10)` 的重试机制，重新生成 UUID 换盐计算。 这意味着，**即使布隆过滤器挂了，我的核心发号业务依然可用，只是短暂退化成了依赖数据库兜底的降级模式**，为我们争取了宝贵的抢修时间。”

#### 🟢 第三步：平滑重建（如何优雅地把数据补回来？）

> “等 Redis 恢复后，我会通过后台的定时任务（如 XXL-JOB）触发数据预热脚本。
>
> **避坑点**：我绝对不会使用全表扫描（`SELECT *`）去捞取这几亿条数据，这会直接把数据库 I/O 打满，引发二次雪崩。 **我的重建方案**：
>
> 1. **游标分批读取**：我会使用基于主键 ID 的游标分页（`SELECT url FROM t_link WHERE id > last_id LIMIT 10000`），每次只拉取一小批数据，极大地减轻了数据库压力。
> 2. **双端切换（Double Buffering）**：我不会直接把数据写回原来的布隆过滤器，因为重建过程中可能有新业务数据产生。我会先在一个**全新的 Redis Key** 上构建完整的布隆过滤器，等几亿条数据全部灌入完毕后，再使用 Redis 的 `RENAME` 命令，在毫秒级的时间内完成新老布隆过滤器的原子切换。
>
> 这样一套组合拳下来，整个灾备恢复过程对 C 端用户是完全无感知的。”

------

这就是高级架构设计的魅力所在：**不盲目相信任何一个中间件，永远在代码里留好防击穿的“退路”（唯一索引 + 异常重试），并且设计好平滑过渡的“手术方案”（主键游标 + 双端切换）。**

**顺着刚才提到的数据库“唯一索引”和“主键 ID 游标”，咱们的短链表 `t_link` 每天都在疯狂增加数据。你是想去深挖一下项目中为了应对海量数据而引入的重量级武器——分库分表组件 ShardingSphere，还是想去看看网关层面的鉴权代码？**



# Ragent

如果你在简历里写了做了这个 RAG 项目，面试官问：“你们的流式对话是怎么做高并发保护的？”

> 你们的流式对话是怎么做高并发保护的？
> 关于流式对话的高并发保护，我们团队没有采用传统的令牌桶或漏桶算法（如 Guava RateLimiter）。因为大模型响应往往需要十几秒，传统限流在超限时通常会直接报错或阻塞等待，这会导致前端体验很差，或者瞬间耗尽 Tomcat 的核心 HTTP 线程池，引发整个服务雪崩。
> 为此，我独立设计并实现了一个基于 Redis 的异步非阻塞排队限流组件。
> 首先，在 AOP 拦截到 SSE 请求后，我们会立刻将执行逻辑封装成异步任务，直接 return 放行，释放 Tomcat 主线程。
> 接着，任务会进入排队机制。我利用 Redis 的 ZSet（有序集合） 和全局递增序号作为 Score，实现了一个分布式的 FIFO 公平队列。同时配合 Redisson 的信号量（Semaphore） 来严格控制与大模型交互的最高并发度。为了防止高并发下的超卖或插队，底层核心的『检查队列排名与获取令牌』逻辑，我是通过编写 Lua 脚本来保证绝对原子性的。
> 在挂起等待阶段，传统的纯轮询太耗性能，纯监听又怕丢消息。所以我借鉴了 Nacos 动态配置的核心思想，采用了 『Redis Pub/Sub 广播 + 定时任务轮询』的推拉结合机制。这既降低了唤醒延迟，又保证了调度的绝对可靠。
> 最后也是最关键的一点，SSE 属于长连接，用户在排队时极有可能中途关闭浏览器。为了兜底，我深度利用了 SseEmitter 的生命周期回调（onCompletion、onTimeout 等），只要连接意外断开，立刻触发回调去 Redis 中移除排队元素并归还令牌，彻底杜绝了分布式死锁和资源泄漏问题。

  - 学习方法：

### 阶段 2：RAG 主链路（对话侧）——`bootstrap/rag`（5–7 天）

**目标**：把“用户问一个问题 → 系统生成回答”这条链路吃透，这是你面试时最核心的“项目讲解主线”。

  - 建议顺序：

    2. **问题重写与拆分**：`core/rewrite/QueryRewriteService` 等

### 阶段 4：模型工程化、`infra-ai`、MCP（3–4 天）

**目标**：把“AI 相关的工程能力”串起来，这部分是你和普通 CRUD 候选人的差异化所在。

  - 关注目录：`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra`

  - 面试点：
