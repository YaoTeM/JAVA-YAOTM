### 二、逐段串联：每一块负责什么、怎么协同？

#### 1）Web 层：`WebAutoConfiguration` + `GlobalExceptionHandler`

**职责：**

- `WebAutoConfiguration` 把 `GlobalExceptionHandler` 注册为 Bean。
- `GlobalExceptionHandler` 统一捕获 Controller 抛出的各种异常（业务异常、ClientException、系统异常等），转成统一的 `Result` 返回结构和错误码。

**你可以这样概括：**

> 我们在 `framework` 里提供了 `WebAutoConfiguration`，自动装配了一个 `GlobalExceptionHandler`。  
> 所有业务模块只要依赖 `framework`，就能统一用这一套异常拦截和返回体规范，避免每个项目都重复写一套。

---

#### 2）用户上下文层：`LoginUser` + `UserContext`

**角色关系：**

```text
LoginUser  = “当前登录用户的一张快照”
UserContext = “基于 TTL 的全局上下文容器”
```

**典型调用流程：**

```text
[拦截器]
  - 根据 token 查询用户
  - 组装 LoginUser(userId, username, role, avatar)
  - UserContext.set(loginUser)

[业务代码任何地方]
  - UserContext.getUserId()
  - UserContext.getRole()
  - UserContext.requireUser()

[请求结束 finally]
  - UserContext.clear()
```

**TransmittableThreadLocal 的意义：**

- 普通 ThreadLocal 在使用线程池时容易丢失上下文；
- Ragent 里大量用线程池做异步检索 / 模型调用，所以用 `TransmittableThreadLocal<LoginUser>`，保证上线文能在子线程中继续使用。

**一段可以背的话：**

> 当前登录用户的信息通过 `LoginUser` 封装，然后存进 `UserContext` 这个基于 `TransmittableThreadLocal` 的容器里。  
> 这样 Controller、Service 甚至异步线程都可以直接 `UserContext.getUserId()` 拿到用户信息，而不需要层层透传参数，请求结束时再 `clear()` 掉避免线程池复用导致脏数据。

---

#### 3）数据库基础设施层：`DataBaseConfiguration` + `MyMetaObjectHandler`

**两个关键 Bean：**

```text
@Bean MybatisPlusInterceptor
    -> addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL))

@Bean MetaObjectHandler myMetaObjectHandler()
    -> new MyMetaObjectHandler()
```

**配合关系：**

- **分页拦截器**：统一处理 MyBatis-Plus 的分页 SQL，自动给分页查询拼 `limit`，并指定数据库类型为 MySQL。
- **自动填充器**：`MyMetaObjectHandler` 在 insert/update 时，自动给实体中标注了 `@TableField(fill = ...)` 的字段赋值，比如：
  - `createTime` / `updateTime`
  - （很可能还会用 `UserContext.getUserId()` 给 `createBy` / `updateBy` 赋值）

你可以用这样一段串起来：

> 数据库层这边，我们在 `DataBaseConfiguration` 里统一注入了 MyBatis-Plus 的分页拦截器和元数据自动填充器。  
> 业务层只要使用 MyBatis-Plus 的分页 API，SQL 会自动被拦截器改写成带 `limit` 的 MySQL 分页语句；  
> 同时，`MyMetaObjectHandler` 会在插入和更新时自动填充创建时间、更新时间，以及结合 `UserContext` 给 `createBy`、`updateBy` 这类审计字段赋值。  
> 这些配置都放在 `framework` 模块里，任何业务模块复用这套配置即可，保证了工程的一致性。

```44:113:D:/codex/ragent/framework/src/main/java/com/nageoffer/ai/ragent/framework/web/GlobalExceptionHandler.java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 1. 参数校验异常 (@Valid / @Validated)
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> validExceptionHandler(HttpServletRequest req, MethodArgumentNotValidException ex) {
        BindingResult bindingResult = ex.getBindingResult();
        FieldError firstFieldError = CollectionUtil.getFirst(bindingResult.getFieldErrors());
        String msg = Optional.ofNullable(firstFieldError)
                .map(FieldError::getDefaultMessage)
                .orElse(StrUtil.EMPTY);

        return Results.failure(BaseErrorCode.CLIENT_ERROR.code(), msg);
    }

    // 2. 项目内主动抛出的三类异常 (AbstractException 子类)
    @ExceptionHandler(AbstractException.class)
    public Result<Void> abstractException(HttpServletRequest req, AbstractException ex) {
        if (ex.getCause() != null) {
            log.error("[{}] {} [ex] {}", req.getMethod(), req.getRequestURL().toString(), ex, ex.getCause());
            return Results.failure(ex);
        }
        // 没有 cause 时，自己拼一个简化版栈信息日志
        ...
        return Results.failure(ex);
    }

    // 3. Sa-Token 未登录异常
    @ExceptionHandler(NotLoginException.class)
    public Result<Void> notLoginException(HttpServletRequest req, NotLoginException ex) {
        log.warn("[{}] {} [auth] not-login: {}", ...);
        return Results.failure(BaseErrorCode.CLIENT_ERROR.code(), "未登录或登录已过期");
    }

    // 4. Sa-Token 无权限异常
    @ExceptionHandler(NotRoleException.class)
    public Result<Void> notRoleException(HttpServletRequest req, NotRoleException ex) {
        log.warn("[{}] {} [auth] no-role: {}", ...);
        return Results.failure(BaseErrorCode.CLIENT_ERROR.code(), "权限不足");
    }

    // 5. 兜底: 任何没被单独处理的 Throwable
    @ExceptionHandler(Throwable.class)
    public Result<Void> defaultErrorHandler(HttpServletRequest req, Throwable throwable) {
        log.error("[{}] {} ", req.getMethod(), getUrl(req), throwable);
        return Results.failure(); // -> SERVICE_ERROR
    }
}
```

### 六、你可以怎么在面试里“一口气讲完这一整块”？

可以按这个顺序组织一段 2–3 分钟的讲解（示例话术）：

> 在 Ragent 这个项目里，我们对接口返回和异常处理做了一套比较完整的工程化设计。  
>
> 首先所有接口都统一用一个 `Result<T>` 来包装返回值，里面包含 `code`、`message`、`data` 和 `requestId`，其中 `"0"` 表示成功，其他 code 都是错误码。我们通过一个 `Results` 工具类来快速构造成功和失败的响应，比如 `Results.success(data)` 或 `Results.failure()`。  
>
> 错误码这块我们定义了一个 `IErrorCode` 接口，然后用 `BaseErrorCode` 枚举实现，按照阿里规范区分 A 类客户端错误、B 类服务端错误、C 类第三方错误，比如 `CLIENT_ERROR`、`SERVICE_ERROR`、`REMOTE_ERROR` 等。  
>
> 在异常层面，我们有一个 `AbstractException` 基类，内部绑定了 `errorCode` 和 `errorMessage`，上层再细分出 `ClientException` 和 `ServiceException` 等不同场景的异常，业务代码里只需要在合适的地方抛出对应异常即可。  
>
> 最后，通过一个 `@RestControllerAdvice` 标注的 `GlobalExceptionHandler` 做全局异常拦截：参数校验失败会被单独拦截并返回客户端错误码；项目内部主动抛出的 `AbstractException` 会被统一转换成携带错误码的 `Result`；Sa-Token 抛出的未登录、无权限异常也有对应的提示；剩下未捕获的 Throwable 则走兜底逻辑，返回一个统一的系统错误码。  
>
> 这样无论是前端还是调用方，只要看 code 就知道错误类型，格式也始终统一，同时日志里保留了足够的异常栈信息方便排查，这一整块在项目中体现了错误码规范化和异常处理的工程化能力。

## 二、分布式 ID：从启动到 insert 的完整链路

### 2.1 雪花 ID 在项目里的分工

```text
  ┌─────────────────────────────────────────────────────────────────┐
  │  雪花 ID 结构（64 bit）                                            │
  │  1bit符号 + 41bit时间戳 + 10bit(5bit datacenterId + 5bit workerId)  │
  │  + 12bit 序列号                                                    │
  └─────────────────────────────────────────────────────────────────┘

  多机部署时：每台机器要有不同的 (workerId, datacenterId)，否则会重复。
  所以需要“中心”分配：这里用 Redis + Lua 分配并持久化。
```

流程可以看成三步：

```text
  [1] 应用启动
        |
        v
  SnowflakeIdInitializer.init()  (@PostConstruct)
        |
        v
  [2] Redis 执行 Lua：分配 (workerId, datacenterId)
        |
        v
  Hutool Snowflake(workerId, datacenterId) 注册到 IdUtil
        |
        v
  [3] 业务 insert 时
        |
        v
  MyBatis-Plus 发现主键为 ASSIGN_ID
        |
        v
  CustomIdentifierGenerator.nextId(entity)
        |
        v
  IdUtil.getSnowflakeNextId()  → 得到 Long 主键
```

也就是说：**谁分配 (workerId, datacenterId)** 在 `SnowflakeIdInitializer`，**谁在 insert 时真正生成 id** 在 `CustomIdentifierGenerator`。

### 2.2 SnowflakeIdInitializer：用 Redis 分配 workerId / datacenterId

```44:72:D:/codex/ragent/framework/src/main/java/com/nageoffer/ai/ragent/framework/distributedid/SnowflakeIdInitializer.java
    @PostConstruct
    public void init() {
        DefaultRedisScript<List> script = new DefaultRedisScript<>();
        script.setScriptSource(new ResourceScriptSource(new ClassPathResource("lua/snowflake_init.lua")));
        script.setResultType(List.class);

        try {
            List<Long> result = stringRedisTemplate.execute(script, Collections.emptyList());

            if (CollUtil.isEmpty(result) || result.size() != 2) {
                throw new RuntimeException("从Redis获取WorkerId和DataCenterId失败");
            }

            Long workerId = result.get(0);
            Long datacenterId = result.get(1);

            Snowflake snowflake = new Snowflake(workerId, datacenterId);
            Singleton.put(snowflake);

            log.info("分布式Snowflake初始化完成, workerId: {}, datacenterId: {}", workerId, datacenterId);
        } catch (Exception e) {
            throw new RuntimeException("分布式Snowflake初始化失败", e);
        }
    }
```

要点：

- `@PostConstruct`：Spring 容器创建好本 Bean 后执行一次。
- 用 **Redis 执行 Lua**：在 Redis 里原子地分配并递增 (workerId, datacenterId)，多实例部署时每个实例拿到不同组合。
- 用 **Hutool 的 Singleton.put(snowflake)**：把当前实例的 Snowflake 放进 Hutool 的全局单例，后面 `IdUtil.getSnowflakeNextId()` 用的就是这个实例。

Lua 逻辑（简化理解）：

```text
  Redis Hash: snowflake_work_id_key
    - dataCenterId: 0..31
    - workId: 0..31

  首次：exists == 0 → 初始化 (0,0)，返回 {0,0}
  之后：workId 先递增到 31，再 dataCenterId 递增，workId 回 0
  → 保证 (workerId, datacenterId) 在 0..31 范围内不重复分配
```

字符画：

```text
  实例 A 启动 ──execute Lua──> Redis 返回 (0,0)   → Snowflake(0,0)
  实例 B 启动 ──execute Lua──> Redis 返回 (1,0)   → Snowflake(1,0)
  实例 C 启动 ──execute Lua──> Redis 返回 (2,0)   → Snowflake(2,0)
  ...
```

这样多机部署时，每台机器的 Snowflake 参数不同，生成的 Long 主键就不会冲突。

```text
  IdUtil.getSnowflakeNextId()  → 例如 1849123456789012345L
        |
        v
  写回 entity.id，再执行 INSERT，主键不再依赖数据库自增
```

所以：**“带 @Valid 的请求体”** 和 **validExceptionHandler** 没关系；**“分布式 ID”** 和 **CustomIdentifierGenerator + SnowflakeIdInitializer** 是绑在一起的：一个负责“算 id”，一个负责“为每台机器分配雪花参数”。

## 三、公共字段自动填充：MyMetaObjectHandler

### 3.1 谁在什么时候被调用

```text
  insert(entity) 或 updateById(entity)
        |
        v
  MyBatis-Plus 在执行 SQL 前回调
        |
        +-- insert  → MetaObjectHandler.insertFill(metaObject)
        +-- update  → MetaObjectHandler.updateFill(metaObject)
```

只有实体里**有字段**且**标注了 `@TableField(fill = ...)`** 的，才会被填充；没标注的不会动。

### 3.2 MyMetaObjectHandler 具体填了哪些字段

```27:40:D:/codex/ragent/framework/src/main/java/com/nageoffer/ai/ragent/framework/database/MyMetaObjectHandler.java
public class MyMetaObjectHandler implements MetaObjectHandler {

    @Override
    public void insertFill(MetaObject metaObject) {
        strictInsertFill(metaObject, "createTime", Date::new, Date.class);
        strictInsertFill(metaObject, "updateTime", Date::new, Date.class);
        strictInsertFill(metaObject, "deleted", () -> 0, Integer.class);
    }

    @Override
    public void updateFill(MetaObject metaObject) {
        strictUpdateFill(metaObject, "updateTime", Date::new, Date.class);
    }
}
```

- **insert 时**：若实体有 `createTime` / `updateTime` / `deleted` 且当前值为空，就填 `new Date()` / `new Date()` / `0`。
- **update 时**：若实体有 `updateTime` 且未设值，就填 `new Date()`。
- `strict*Fill`：只有该属性**当前为空**时才填，避免覆盖业务已经设好的值。

和 DO 的对应关系（以 KnowledgeBaseDO 为例）：

```70:79:D:/codex/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/knowledge/dao/entity/KnowledgeBaseDO.java
    @TableField(fill = FieldFill.INSERT)
    private Date createTime;

    @TableField(fill = FieldFill.INSERT_UPDATE)
    private Date updateTime;

    @TableLogic
    private Integer deleted;
```

```text
  字段            注解                        insert 时              update 时
  createTime     FieldFill.INSERT            MyMetaObjectHandler 填  不填
  updateTime     FieldFill.INSERT_UPDATE      MyMetaObjectHandler 填  填
  deleted        @TableLogic + Handler 填     MyMetaObjectHandler 填 0  不填（逻辑删除用 @TableLogic 语义）
```

注意：`createdBy` / `updatedBy` 在当前 `MyMetaObjectHandler` 里**没有**自动填，是业务在 Service 里用 `UserContext.getUsername()` 之类手动设的（例如 KnowledgeBaseServiceImpl 里对 `createdBy`/`updatedBy` 的赋值）。若要统一成“公共字段”，也可以在 Handler 里用 `UserContext.getUserId()` 做自动填充（需要防 NPE）。

## 四、DO 注解小结（面试可背）

| 注解                                          | 作用                                 | 谁用                                               |
| --------------------------------------------- | ------------------------------------ | -------------------------------------------------- |
| `@TableId(type = IdType.ASSIGN_ID)`           | 主键不由 DB 自增，由应用生成         | CustomIdentifierGenerator（雪花）                  |
| `@TableField(fill = FieldFill.INSERT)`        | 仅 insert 时自动填                   | MyMetaObjectHandler.insertFill                     |
| `@TableField(fill = FieldFill.INSERT_UPDATE)` | insert 和 update 都自动填            | MyMetaObjectHandler.insertFill / updateFill        |
| `@TableLogic`                                 | 逻辑删除字段，查询时自动加 deleted=0 | MyBatis-Plus 查询；插入时若需默认 0 可交给 Handler |

## 五、面试可用的简短表述

- **分布式 ID**：我们不用数据库自增，用雪花。应用启动时 `SnowflakeIdInitializer` 通过 Redis 执行 Lua 脚本原子分配 (workerId, datacenterId)，注册到 Hutool 的 Snowflake；insert 时 MyBatis-Plus 对 `@TableId(ASSIGN_ID)` 调用 `CustomIdentifierGenerator.nextId()`，内部用 `IdUtil.getSnowflakeNextId()` 生成 Long 主键，多实例下由 Redis 保证不同实例拿到不同雪花参数，避免主键冲突。
- **公共字段**：`DataBaseConfiguration` 里注册了自定义的 `MyMetaObjectHandler`。insert 时自动填 `createTime`、`updateTime`、`deleted`（0）；update 时自动填 `updateTime`。实体上用 `@TableField(fill = FieldFill.INSERT)` / `INSERT_UPDATE` 声明需要自动填的字段，这样业务代码不用每次手写 set 时间戳和删除标记。

如果你愿意，下一步可以选一个具体的 `insert` 调用（比如创建知识库），从 `knowledgeBaseMapper.insert(kbDO)` 一直跟到 SQL 生成，把「主键是谁赋的、createTime/updateTime 是谁赋的」对到代码行，方便你面试时举一个具体例子。
